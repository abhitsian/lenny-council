## One-liner

Eliminate ambiguity at the project level through shared vocabulary, single-source-of-truth docs, and real-time visual decision-making — so that disagreements are about actual trade-offs, not misunderstandings.

## The framework

Naomi calls this "canonical everything." It has five components, each with a concrete mechanic:

**1. Canonical nomenclature**
Write down every key term and its definition. Different words for the same thing — or worse, the same word for different things — produces conversations where people talk past each other. Fix the vocabulary first.

**2. Canonical doc**
Every project has exactly one doc that any newcomer can land on to understand the project. That doc contains: work streams, single-threaded owners (one per work stream, recursing into sub-work streams), process by which the team works together, links to everything else. Not hundreds of docs — one canonical doc that links to all the others.

**3. Canonical meetings and channels**
One canonical meeting, one canonical email list, one canonical chat group per working team. No re-inventing the audience with slightly different permutations each time. Pairwise conversations that duplicate context are waste.

**4. Canonical visuals**
Tables beat flat text. When evaluating options: three options as rows, evaluation criteria as columns (functional perspective or optimization criteria), color-coded red/yellow/green. Real-time edit the projected visual during the meeting so everyone is looking at the same thing, decisions are explicit, and next steps are written down before people leave the room. Always numbered lists, never bullets — you can say "feedback on item three," you can't point to "the second bullet."

**5. Canonical meeting cadence**
Send pre-read 24 hours before. Document decisions and next steps within 24 hours after, via reply-all to the original calendar invite (the invite is the canonical unit for pre- and post-meeting communication). Come with three options and a recommendation — never a binary choice, never a pros-and-cons list.

**The school pyramid (simplification layer)**
When joining a complex project already running at PhD level, build the curriculum from kindergarten up — identify the most basic building blocks, then layer complexity on top. This is not oversimplification; it is revealing what the real disagreements are.

**80/20 diagnosis rule**
When a project is stuck, 80% of the time the issue is people or process, not strategy. Fix execution first. Only when a project is executing perfectly can you determine whether the strategy is right or wrong.

## When it applies / when it doesn't

**Applies when:**
- You are joining a project mid-stream and need to ramp quickly on gnarly cross-functional work
- A team is meeting frequently but leaving without clear decisions or owners
- Multiple functions are using different terms for the same things
- There is ambiguity about who owns what

**Does not apply when:**
- The team is small (two to three people), co-located, and already sharing context naturally — the overhead of canonical docs adds friction without value
- The project is exploratory/discovery-phase — canonical docs assume enough is known to define work streams and owners; early-stage work needs looser structure

**Critical boundary:** Extreme clarity is not process for process's sake. Naomi is explicit that the goal is to move faster and hold fewer meetings. If the process is slowing you down, something is misconfigured.

## Signature questions

1. "Where is the canonical doc? If five people on this project answer that differently, we have a clarity problem before we have a strategy problem."
2. "For each work stream — who is the single-threaded owner? Not 'the team.' One name."
3. "Are we disagreeing on the facts, or on the trade-offs? Let's separate those two conversations."
4. "Have we defined our key terms? Can I see the canonical nomenclature for this project?"
5. "What are the three options, and what's the recommendation? Walk me through the traffic-light table."

## Common misuse

Naomi explicitly warns against mistaking this framework for bureaucracy: "The process is not for process's sake, it's ultimately to help us all move faster and work better." The failure mode is treating canonical docs and canonical meetings as overhead or compliance checkboxes. The point is speed — less time in duplicative conversations, fewer misunderstandings to unwind. If a team adopts the forms (the doc template, the color-coded table) without the underlying goal of reducing ambiguity, they get all the cost and none of the benefit.

A second misuse: relying entirely on people-and-process facilitation without developing a first-party opinion. Naomi identifies this as her own growth edge: "A PM cannot outsource their perspective or delegate their thinking through people and process." The conductor runs the orchestra, but the conductor also needs to know what the music should sound like.
